# Session Runner: Replace OpenCode with Direct Claude Code Agent Execution

**Date:** 2026-03-20
**Status:** Draft

## Problem

The tracker orchestrator dispatches work items to Claude Code via OpenCode, which proxies through a Claude Max proxy server. This 5-process chain (Tracker → OpenCode → Claude Code → Proxy → Anthropic API) causes:

- Timeout failures from proxy queue starvation (PQueue concurrency:1 serializes all API calls)
- Fragile session monitoring via OpenCode's SSE event stream
- Unnecessary complexity — three intermediary services for what should be a direct connection

## Solution

Replace OpenCode and the proxy with a **session runner**: a lightweight child process that uses the Claude Agent SDK to run Claude Code directly. Communication between the orchestrator and runner uses stdio JSON pipes. The tracker dashboard provides a native session viewer with live activity feed and steering input.

**Before (5 processes):**
```
Tracker → OpenCode → Claude Code → Proxy → Anthropic API
```

**After (2 processes):**
```
Tracker → Runner (child process, uses Agent SDK → Claude Code) → Anthropic API
```

## Design

### 1. Session Runner (`src/session-runner.ts`)

A standalone Node.js script spawned by the orchestrator as a child process. One runner per active session.

**SDK dependency:** `@anthropic-ai/claude-agent-sdk` (version ^0.2.25, published on npm). Already in use at `/Users/martin/opencode-claude-max-proxy`. The SDK's `query()` function returns a `Query` async generator with methods including `streamInput()` for injecting user turns mid-conversation. API verified against the SDK's TypeScript declarations at `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts`.

**Lifecycle:**

1. Reads config JSON from stdin on startup
2. Calls `query()` from `@anthropic-ai/claude-agent-sdk`
3. Writes structured JSON events to stdout (one per line)
4. Listens on stdin for steering/abort messages
5. Exits when the agent finishes (exit code 0 = success, 1 = failure)

**SDK configuration:**

| Option | Value |
|--------|-------|
| `cwd` | Project's `working_directory` |
| `permissionMode` | `'bypassPermissions'` |
| `allowDangerouslySkipPermissions` | `true` |
| `includePartialMessages` | `true` |
| `model` | From `CODER_MODEL_ID` config |
| `systemPrompt` | `{ type: 'preset', preset: 'claude_code', append: '<security rules>' }` |
| `abortController` | Created on startup, triggered by stdin abort event |
| `maxTurns` | Configurable, default 100 |
| `persistSession` | `true` (enables resume after interruption) |

**Stdin protocol (orchestrator → runner):**

Initial config (first line):
```json
{
  "event": "config",
  "itemKey": "TRACK-42",
  "prompt": "Full prompt text with item context, description, comments...",
  "systemPromptAppend": "Security rules: do not modify blocked files...",
  "cwd": "/Users/martin/myproject",
  "model": "opus",
  "maxTurns": 100,
  "attachments": [
    {"path": "/absolute/path/to/image.png", "mime": "image/png", "filename": "screenshot.png"}
  ]
}
```

Steering message (human intervention):
```json
{"event": "steer", "message": "Wrong file — check src/auth/middleware.ts instead"}
```

Abort:
```json
{"event": "abort"}
```

**Stdout protocol (runner → orchestrator):**

All events are newline-delimited JSON:

```json
{"event": "started", "sessionId": "runner_<random-hex>", "pid": 12345}
{"event": "tool_use", "tool": "Edit", "file": "src/auth.ts", "elapsed": 2.1}
{"event": "tool_result", "tool": "Edit", "status": "success"}
{"event": "text", "content": "Fixed the token validation logic..."}
{"event": "status", "status": "compacting"}
{"event": "heartbeat", "elapsed": 45, "turns": 3}
{"event": "completed", "result": "success", "duration": 120, "turns": 8}
{"event": "error", "message": "Rate limited", "recoverable": false}
```

**Session ID:** Generated by the runner on startup as `runner_<random-hex>` (24 chars). The SDK may also provide an internal session ID via the init message; if so, it is captured and stored for potential session resume, but the runner-generated ID is canonical for the `activeSessions` map and execution audits.

**Heartbeat interval:** Every 30 seconds. Critical for preventing stale session detection — the orchestrator uses `lastActivityAt` to detect stuck sessions. During long tool calls (e.g., test suites), the SDK may not emit events for minutes; the heartbeat ensures liveness.

**Event mapping from SDK message types:**

| SDK Message Type | Runner Event |
|-----------------|--------------|
| `SDKSystemMessage` (init) | `started` (captures sessionId) |
| `SDKToolProgressMessage` | `tool_use` |
| `SDKToolUseSummaryMessage` | `tool_result` |
| `SDKAssistantMessage` (text blocks) | `text` |
| `SDKResultMessage` (success) | `completed` |
| `SDKResultMessage` (error) | `error` |
| `SDKStatusMessage` (compacting) | `status` |
| periodic timer | `heartbeat` |

**Steering via `streamInput`:**

The SDK's `query()` returns a `Query` object with a `streamInput()` method that accepts an async iterable of `SDKUserMessage`. When a steering message arrives on stdin, the runner creates a user message and feeds it through `streamInput`. The agent receives it as the next user turn and adjusts its behavior.

**Stderr:** The runner passes the Agent SDK's subprocess stderr through to its own stderr, allowing the orchestrator to capture debug logs separately from the structured event stream.

### 2. Orchestrator Integration

**Config (`src/config.ts`):**

New environment variable:
```
DISPATCH_MODE=runner|opencode    # default: "opencode"
```

When `DISPATCH_MODE=runner`, the orchestrator uses the runner dispatch path. All existing OpenCode config vars are ignored but remain functional for `DISPATCH_MODE=opencode`.

**Dispatch path (`src/orchestrator.ts`):**

New function `dispatchViaRunner(item, project)`:

1. Builds prompt using existing `buildPrompt()` (unchanged)
2. Splits prompt into system prompt append (security rules) + user prompt (item context)
3. Spawns runner: `node dist/session-runner.js` as child process
4. Writes config JSON to child stdin
5. Reads stdout line-by-line, parses JSON events
6. Routes events into the existing session state machine:

| Runner Event | Orchestrator Action |
|-------------|-------------------|
| `started` | `setSessionInfo(itemId, sessionId, "running")`, store PID in `ActiveSession` |
| `completed` | `handleSessionComplete(sessionId)` — existing handler, unchanged |
| `error` | `handleSessionError(sessionId, message)` — existing handler, unchanged |
| `tool_use`, `text`, `heartbeat` | Update `lastActivityAt` on `ActiveSession` (prevents stale detection) |
| `tool_use`, `text` | Buffer event for dashboard viewer |

7. Stores child process reference on `ActiveSession` for abort/cleanup

**What remains unchanged:**

- `buildPrompt()`, `buildResearchPrompt()` — same prompt construction
- `handleSessionComplete()` — same completion logic (state advance, scheduled task recycling, etc.)
- `handleSessionError()` — same error logic (circuit breaker, per-item failures, etc.)
- `activeSessions` map — same structure, PID field now holds runner PID
- Stale session detection — `isProcessAlive(pid)` checks runner PID instead of OpenCode PID
- Approval/comment/clarification watchers — unchanged
- Execution audit recording — unchanged
- Concurrency limits (`OPENCODE_MAX_CONCURRENT`, `OPENCODE_MAX_PER_PROJECT`) — renamed to generic names but same logic

**What's removed (in runner mode only):**

- OpenCode SDK client creation (`createOpencodeClient`)
- SSE event stream (`startEventStream`, `handleSseEvent`)
- OpenCode project ID resolution (`resolveOpencodeProjectId`)
- Compaction/413 recovery state machine (SDK handles internally)
- Permission waiting state / TRACK-203 handling (bypassed)

**What's new:**

- `dispatchViaRunner()` function
- Child process lifecycle management (spawn, pipe setup, cleanup)
- Runner event stream parser
- Event buffering for dashboard viewer

**Abort flow:**

1. Sends `{"event":"abort"}` to runner's stdin
2. Runner triggers `abortController.abort()` on the SDK query
3. If runner doesn't exit within 5 seconds, orchestrator sends SIGTERM
4. If still alive after 5 more seconds, SIGKILL
5. Cleanup: remove from `activeSessions`, update DB status

**Research dispatch:**

`dispatchForClarificationViaRunner()` follows the same pattern as `dispatchViaRunner()` but with the research prompt (from existing `buildResearchPrompt()`). The runner doesn't need to know the difference — it just runs whatever prompt it receives.

**Session recovery after tracker restart:**

When the tracker process restarts, all runner child processes receive SIGTERM from the graceful shutdown handler and exit. The stdio pipes are severed. On restart, the orchestrator must clean up:

1. On startup, scan the DB for items with `session_status = 'running'` or `'pending'`
2. For each, check if the runner PID is still alive via `isProcessAlive(pid)`
3. If the process is dead (expected after restart): clear session info, unlock the item, add a comment noting the interruption
4. The item stays in its current state (e.g., `in_development`) and will be re-dispatched on the next tick if still `approved` — or moved back to `approved` if it was in `in_development` with no meaningful work done
5. If the SDK's `persistSession` was true, the session ID is stored. A future enhancement could resume interrupted sessions via `options.resume` instead of starting fresh. This is not in scope for v1 — clean restart is the initial behavior.

The existing `recoverActiveSessions()` function handles this for OpenCode mode. The runner mode equivalent follows the same pattern but is simpler: no SDK status polling needed, just PID liveness checks and DB cleanup.

### 3. Dashboard Session Viewer

**New API endpoints:**

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/items/:id/session/events` | SSE stream of runner events for a live session |
| `POST` | `/api/v1/items/:id/session/steer` | Send a steering message to the running agent |

The existing `POST /api/v1/items/:id/session/abort` endpoint continues to work.

**Event buffering:**

Each `ActiveSession` gains an `events: RunnerEvent[]` array, capped at 200 entries. Events from the runner's stdout are:
1. Processed by the state machine (completion, error, heartbeat)
2. Appended to the buffer
3. Pushed to any connected SSE clients

When a browser connects to the SSE endpoint:
1. Receives the buffered events as a burst (catch-up)
2. Then receives live events as they arrive
3. Connection closes when the session ends

Each SSE event includes a monotonic `id` field. On reconnection, the browser's `EventSource` sends `Last-Event-ID`, and the server replays only events after that ID from the buffer. This handles brief network drops without re-sending the full buffer.

**Existing endpoint update:**

The `GET /api/v1/items/:id/session` endpoint currently returns `opencode_url`. In runner mode, it omits this field and instead returns `{ dispatch_mode: "runner", session_id, status, event_count }` so the dashboard knows to render the native session viewer instead of an OpenCode deep link.

**Dashboard UI (`src/ui/core.html`):**

When the user opens an item with an active runner session, the detail panel shows:

- **Activity feed:** Scrolling log of events, color-coded by type:
  - Tool use: tool name + file path
  - Agent text: the agent's reasoning/explanation
  - Errors: red, with message
  - Heartbeat: subtle elapsed time indicator
- **Steering input:** Text field at the bottom. On submit, POSTs to the steer endpoint. The message appears in the feed as a "human" entry.
- **Abort button:** Red button, calls the existing abort endpoint.
- **Status indicator:** Running/completed/failed badge.

The activity feed is plain and functional — styled consistently with the existing dashboard aesthetic. No terminal emulator, no xterm.js. Just a scrolling div with formatted event entries.

**UI section:** Added under `// -- Session Viewer --` section header in `core.html`, following the project's section header convention.

**Security note — `BLOCKED_PATHS` enforcement:** In runner mode with `permissionMode: 'bypassPermissions'`, there is no filesystem-level enforcement of blocked paths. Blocked paths remain prompt-level enforcement only (soft enforcement via system prompt instructions), same as the current OpenCode mode. The agent is instructed not to modify these files; it is not physically prevented. This is acceptable because the same trust model applies today — OpenCode's coder sessions also rely on prompt-level enforcement.

### 4. Agent Configuration

**System prompt:**

The runner uses the Agent SDK's system prompt append mode. The existing `buildPrompt()` is refactored to return a `{ systemAppend: string, userPrompt: string }` object instead of a single string. A new `buildPromptParts()` function provides this, while `buildPrompt()` remains for backwards compatibility (concatenates both parts for OpenCode mode).

**Split boundary:**

- **System prompt append** (stable across all items, rarely changes):
  - Blocked file patterns from `BLOCKED_PATHS`
  - "Do not create new tracker items or approve items"
  - "Do not modify tracker infrastructure"
  - "Do not impersonate human actors"
  - General security rules
- **User prompt** (item-specific, changes per dispatch):
  - Item key, title, project name, priority, assignee, labels
  - Description (the task spec)
  - Comments (with pre/post-approval segregation)
  - Dependency/blocker information
  - Workflow instructions ("move to in_review when done", research-mode instructions, etc.)
  - Space-specific data (scheduled task config, etc.)

The boundary is: security constraints go in the system prompt (they're enforced regardless of the task), workflow instructions go in the user prompt (they're task-specific).

**Agent config validation (runner mode):**

`validateAgentConfig()` in runner mode checks:
- The Claude Code executable is resolvable (SDK can find it)
- The `session-runner.ts` build artifact exists at `dist/session-runner.js`

The OpenCode agent file check (`~/.config/opencode/agents/tracker-worker.md`) is skipped.

**Model configuration:**

The existing `CODER_MODEL_ID` config maps to the SDK's model option. In runner mode, `CODER_MODEL_PROVIDER` is ignored (Claude Code authenticates directly via Max subscription, not via API key + provider).

### 5. New Dependency

Add `@anthropic-ai/claude-agent-sdk` to the tracker's `package.json`. This is the only new dependency. The SDK is already used in the proxy project and is well-tested.

### 6. Backwards Compatibility

All OpenCode integration code remains in the codebase, gated behind `DISPATCH_MODE=opencode` (the default). Users of the public tracker project who use OpenCode are unaffected. The runner mode is opt-in via `DISPATCH_MODE=runner`.

Config variables that apply to both modes:
- `ORCHESTRATOR_ENABLED`, `ORCHESTRATOR_INTERVAL`
- `OPENCODE_MAX_CONCURRENT`, `OPENCODE_MAX_PER_PROJECT` (may rename to generic names with OpenCode names as aliases)
- `SESSION_TIMEOUT`
- `CIRCUIT_BREAKER_THRESHOLD`, `CIRCUIT_BREAKER_WINDOW`
- `ITEM_DISPATCH_FAILURE_LIMIT`
- `CODER_MODEL_ID`

Config variables only used in opencode mode:
- `OPENCODE_SERVER_URL`, `OPENCODE_PUBLIC_URL`
- `CODER_MODEL_PROVIDER`

Config variables only used in runner mode:
- (none initially — the runner inherits all config from the orchestrator)

## Future Considerations

- **Web UI evolution:** The dashboard session viewer starts as a plain activity feed but can grow into a rich interface with action buttons, cost tracking, tool call inspection, and approval workflows — all within the tracker dashboard.
- **Session resume:** If a runner session is interrupted (tracker restart), the persisted session ID could be used to resume via the SDK's `resume` option.
- **Remote runners:** The stdio protocol could be adapted to run over a network transport (WebSocket, SSH) for distributed execution. Not needed now but the protocol design supports it.
